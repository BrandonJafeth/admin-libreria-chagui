## Architecture

**Stack:** React 19 + TypeScript 6 + Vite 8 + TanStack Router (file-based) + TanStack Query v5 + Supabase + Zustand + shadcn/ui + Tailwind CSS v4. Single-tenant admin dashboard, no auth role distinction — any authenticated Supabase user has full write access.

**Entry:** `src/main.tsx` (Vite SPA). No SSR. `index.html` in project root.

**Routing:** File-based via `src/routes/`. `routeTree.gen.ts` is auto-generated by the Vite plugin on `pnpm dev` — never edit manually. Route structure:
- `/login` → public, redirects to `/` if session exists
- `/_authenticated` → pathless layout with `beforeLoad` auth guard (redirects to `/login`), wraps children in `DashboardLayout`
- `/_authenticated/` → dashboard home (summary cards)
- `/_authenticated/productos/` → product list with filters
- `/_authenticated/productos/nuevo` → create product
- `/_authenticated/productos/$productId` → edit product (loader prefetches via `ensureQueryData`, component uses `useSuspenseQuery`)
- `/_authenticated/categorias/` → category list

**Auth flow:** `useAuth` in `__root.tsx` initializes Supabase session + subscribes to `onAuthStateChange`. Stores state in Zustand (`auth-store`). Root shows spinner while `isLoading`. `_authenticated.tsx` `beforeLoad` does a direct `supabase.auth.getSession()` check — redundant but necessary since `beforeLoad` runs server-side in some contexts and needs a fresh session.

**Data layer:** TanStack Query for all Supabase data. Zustand only for session (`auth-store`) and UI state (`ui-store` — sidebar collapsed). No server state in Zustand.

**Feature layout:**
```
src/features/
├── auth/         → store, hooks, api, components (LoginForm)
├── categories/   → api, hooks (useCategories, mutations), components (Table, Form)
└── products/     → api, hooks (useProducts, useProduct, mutations), components (Table, Form, ImageUploader, ColorPicker)
```

**Products API:** Uses Supabase embed (`.select('*, product_images(...), product_categories(...)')`) rather than a SQL view — simpler, no DB access needed, acceptable for admin workloads. List query filters `es_principal` client-side.

**product_categories N:N:** On update, delete-all then insert-new. Simple, avoids upsert complexity.

**Image uploads:** Supabase Storage bucket `product-images` (must be public). Create with: `supabase storage create product-images --public`. DB trigger `enforce_single_main_image` handles unsetting the previous principal image.

**Path alias:** `@/*` → `src/*` (both tsconfig and vite.config.ts).

## Styling

Tailwind CSS v4. No `tailwind.config.js` — theme is configured via `@theme` in `src/styles.css`. Key tokens:
- `bg-accent` / `text-accent` → primary brand color (red-orange)
- `bg-accent-2` / `bg-accent-3` → secondary brand colors
- `font-heading` → Poppins, `font-body` → Inter
- `bg-primary` / `text-primary-foreground` → mapped to accent (shadcn compatibility)
- `.card-solid` → opaque card with subtle shadow (non-glass surface)
- `.liquid-glass` / `.liquid-glass-strong` → frosted glass surfaces
- `.cuaderno-bg` → signature grid texture

## Form Validation

**Stack:** react-hook-form v7 + Zod v4 + `@hookform/resolvers` v5.

**Critical — use `standardSchemaResolver`, NOT `zodResolver`:**

```ts
import { standardSchemaResolver } from '@hookform/resolvers/standard-schema'

useForm({ resolver: standardSchemaResolver(schema) })
```

**Why not `zodResolver`:** Vite bundles `zod` (app import) and `zod/v4/core` (resolver internal import) as separate modules. The `zodResolver` from `/zod` path uses `error instanceof z4.$ZodError` to detect Zod v4 errors — this fails across module boundaries even though it's the same class. Result: ZodError thrown as unhandled promise rejection, no field errors shown in UI. `standardSchemaResolver` avoids this entirely — it calls `schema['~standard'].validate()` (Standard Schema protocol, which Zod v4 implements natively), no `instanceof` check needed.

**Forms that use this pattern:** `ProductForm`, `CategoryForm`, `UsersTable`.

**Zod schema notes:**
- Use `z.coerce.number({ error: 'message' })` for number inputs (v4 syntax; v3 used `invalid_type_error`)
- Error messages go in `.min(1, 'msg')` / `.max(n, 'msg')` as usual
- `category_ids: z.array(z.string()).min(1, 'msg')` for multi-select chip fields

**ProductForm specifics:**
- `mode: 'onBlur'` — validates on blur before first submit, re-validates on change after first failed submit
- `handleInvalid` sets `validationFailed: true` → shows red banner at top
- `FieldError` component renders `<AlertCircle>` + message below each field
- `category_ids` uses `setValue('category_ids', next, { shouldValidate: true })` so chip toggles trigger validation immediately after first submit attempt
